# 범용 코딩 컨벤션 (JYP)

언어를 불문하고 모든 프로젝트에 적용하는 공통 규칙.
**단, 해당 언어의 커뮤니티 표준(PEP 8, Effective Go, ESLint 권장 등)이나 기존 코드베이스의 스타일과 충돌하면 그쪽을 우선한다.** 이 문서는 언어 표준이 정하지 않는 부분을 채우는 기본값이다.

## 1. 네이밍

- 이름만 보고 역할을 알 수 있게 짓는다. 축약 금지: `mgr`, `tmp`, `dat`, `proc` ✗ → `manager`, `temporary_file`, `user_data` ○
- 함수/메서드는 동사로 시작: `get_user()`, `validateInput()`, `send_report()`
- boolean은 `is_`, `has_`, `can_`, `should_` 접두사: `is_active`, `hasPermission`
- 상수는 UPPER_SNAKE_CASE (언어 관례가 다르면 언어 관례)
- 컬렉션은 복수형: `users`, `error_messages`
- 같은 개념에는 프로젝트 전체에서 같은 단어를 쓴다 (`fetch`/`get`/`load`/`retrieve` 혼용 금지 — 하나 정해서 통일)

## 2. 함수와 파일 크기

- 함수는 한 가지 일만 한다. 40줄을 넘으면 분리를 검토한다.
- 중첩(들여쓰기)이 3단계를 넘으면 early return이나 함수 추출로 평탄화한다.
- 파일이 300줄을 넘으면 책임 분리를 검토한다. (기계적으로 쪼개지 말고 응집도 기준으로)
- 매개변수가 4개를 넘으면 객체/구조체로 묶는 것을 검토한다.

## 3. 폴더/모듈 구조 원칙

- 기능(도메인) 단위로 묶는 것을 우선한다. 계층(controller/service/model) 단위 분리는 규모가 커졌을 때.
- 순환 의존 금지. A가 B를 import하면 B는 A를 import하지 않는다.
- 진입점(main, index)은 얇게 유지한다 — 조립과 실행만 하고 로직을 두지 않는다.
- 공용 유틸은 `utils` 하나에 다 넣지 말고 목적별로 나눈다 (`date_utils`, `file_utils`).

## 4. 에러 처리

- 실패할 수 있는 지점(파일 IO, 네트워크, 외부 입력, 파싱)은 반드시 명시적으로 처리한다.
- 에러를 조용히 삼키지 않는다. 최소한 로그를 남기고, 복구 불가능하면 위로 전파한다.
- 에러 메시지에는 맥락을 담는다: "실패" ✗ → "설정 파일을 읽지 못함: config.yaml (권한 없음)" ○
- 사용자에게 보이는 에러와 개발자용 로그를 구분한다.

## 5. 주석과 문서화

- 설명 주석을 적극적으로 단다. 코드 블록·섹션의 역할과 동작, 그리고 "왜 이렇게 했는지"를 주석으로 남겨 다음 사람이 코드만 읽고 헤매지 않게 한다.
- **모든 새 파일 맨 위에 파일 헤더 주석**을 단다. 작성자명은 항상 `[박진영]` 고정 (`[JYP]` 형태 금지):

```js
/**
 * 파일명 : xxx.js
 * 용도 : ...
 * 최초등록 : YYYY-MM-DD [박진영]
 * 수정일자 : YYYY-MM-DD [박진영] - 변경 이유
 * 수정사항 : ...
 */
```

- 함수 인자·컴포넌트 props 설명은 **선언부 상단 doc comment**(JSDoc `@param`, docstring 등)로 쓴다: 타입, 동작, 기본값, 트리거 시점. 신규 인자/prop 추가 시 doc comment 한 줄을 반드시 함께 추가.
- 구조분해(destructure) 인자 옆 인라인 설명 주석은 금지 — 허용되는 것은 `// ===== 그룹명 =====` 같은 구획 라벨뿐. 개별 설명은 doc comment로.
- 주석 처리된 죽은 코드를 남기지 않는다. 필요하면 git 히스토리에서 찾는다.
- TODO를 남길 때는 `TODO(날짜): 내용` 형식으로 쓴다.

## 6. 설정과 비밀값

- 경로, URL, API 키, 매직 넘버는 하드코딩하지 않는다 → 상수, 설정 파일, 환경 변수로 분리.
- 비밀값(.env, 키 파일)은 절대 커밋하지 않는다. `.env.example`로 필요한 키 목록만 공유한다.
- 기본값은 안전한 쪽으로 (디버그 모드 off, 최소 권한).

## 7. 테스트

- 자동화 테스트의 전략(무엇을 테스트하나)·도구 표준·게이트 규칙은 **`testing.md`가 단일 출처** — 이 문서에서는 중복 정의하지 않는다.

## 8. 의존성

- 표준 라이브러리로 충분하면 외부 패키지를 추가하지 않는다.
- 새 의존성 추가 기준: 활발히 유지보수되는가, 대안 대비 이득이 명확한가. 추가 전 사용자에게 보고.
- 버전은 잠근다 (lock 파일 커밋).

## 9. 규칙 문서화 (메타 규칙)

프로젝트 규칙 문서(CLAUDE.md 등)에 새 규칙을 추가할 때:

- ✅(올바른 예) / ❌(금지 예) 대비 코드 예시를 함께 쓴다.
- 근거를 병기한다 — 어떤 버그·사고에서 나온 규칙인지. (예: "status === 200 단독 비교 금지 — createdResponse는 201이라 신규 등록이 성공 처리 안 되는 버그 반복")
- 강제 수준을 마커로 표시한다: STRICT(위반 금지) / MANDATORY(필수 적용). 확정 날짜(YYYY-MM-DD)를 붙인다.
- 기존 규칙과 겹치면 새 항목을 만들지 말고 기존 항목을 갱신한다.
