# DB 마이그레이션 컨벤션 (JYP)

스키마·데이터 변경을 안전하게 운영 DB에 반영하기 위한 규칙. DB를 쓰는 모든 프로젝트에 적용한다.

## 1. 파일 규칙

- 위치: 프로젝트 루트 `migrations/` 폴더.
- 파일명: **3자리 순번 + 설명** — `001_create_user_table.sql`, `002_add_status_to_order.sql`
- **하나의 마이그레이션 = 하나의 목적.** 여러 변경을 한 파일에 몰아넣지 않는다.
- **적용된 마이그레이션 파일은 절대 수정 금지** — 고칠 게 있으면 새 번호의 파일로 만든다 (근거: 이미 적용된 환경과 새로 적용하는 환경의 스키마가 달라진다).
- 번호 충돌 방지: 새 마이그레이션 작성 전 `migrations/`의 최신 번호를 확인하고, 브랜치 병합 시 번호 중복 여부를 재확인한다 (근거: 병렬 작업에서 같은 번호를 잡으면 적용 순서가 꼬인다).

## 2. 멱등성 (STRICT)

운영 적용 스크립트는 **재실행해도 안전**해야 한다 — 어디까지 적용됐는지 불확실한 상황에서 다시 돌릴 수 있어야 한다.

```sql
/* 001_add_status_to_order : 주문 테이블에 상태 컬럼 추가 */
-- ✅ 멱등 가드
ALTER TABLE orders ADD COLUMN IF NOT EXISTS status VARCHAR(20) NOT NULL DEFAULT 'PENDING';

-- ✅ 시드 데이터는 NOT EXISTS 가드
INSERT INTO code_table (code, name)
SELECT 'NEW_CODE', '신규 코드'
WHERE NOT EXISTS (SELECT 1 FROM code_table WHERE code = 'NEW_CODE');
```

- `IF NOT EXISTS`를 지원하지 않는 구문은 사전 존재 확인 절차를 주석으로 명시한다.

## 3. DDL / DML 분리 (STRICT)

- **DDL(CREATE/ALTER/DROP)과 DML(INSERT/UPDATE/DELETE)은 파일을 분리한다.**
- 근거: MySQL/MariaDB에서 DDL은 **암묵 커밋(implicit commit)** 을 일으켜, 한 파일에서 트랜잭션으로 감싸도 DDL 시점에 이미 커밋된다 — "실패하면 전체 롤백"이라는 기대가 조용히 무너진다.
- DML 파일만 트랜잭션으로 감싼다. DDL 파일은 문장 단위 멱등 가드로 안전성을 확보한다.

## 4. 파괴적 변경은 2단계 배포

컬럼 삭제·이름 변경·타입 축소 등 **되돌리기 어려운 변경**은 한 번에 하지 않는다. **API 응답에 실리는 컬럼이면 이는 DB 변경이 아니라 API 계약 파괴 변경이다** (api.md 4절 — 컬럼명이 곧 필드명이므로 클라이언트가 함께 깨진다):

1. **1차 배포**: 새 구조 추가 + 코드가 새 구조를 사용하도록 전환 (구 컬럼은 미사용 상태로 유지)
2. **2차 배포**: 운영에서 문제없음을 확인한 뒤 구 구조 삭제

- 즉시 삭제가 불가피하면 롤백 스크립트(복구용 SQL)를 같은 번호로 동반 작성한다 (`002_rollback.sql`).

## 5. 적용 기록과 canonical DDL 동기화

- **원칙 (STRICT, 스택 무관)**: 마이그레이션은 **적용 이력을 자동 기록하는 러너로만** 실행한다 — 어떤 파일이 어느 환경에 언제 적용됐는지가 이력 테이블에 남아야, 환경 간 "어디까지 적용됐는지"가 어긋나지 않는다. DB 툴에서 SQL을 수동 실행하지 않는다.
- **스택별 러너 표준** (원칙은 위 하나, 도구만 다르다):

| 스택 | 러너 | 이력 테이블 | 파일명 |
|---|---|---|---|
| Node/Express | 커스텀 러너 `npm run migrate` — `migrations/`를 순번대로 읽어 미적용분만 실행·기록 (스캐폴드가 생성) | `schema_migrations` (표준 스키마 00번) | `001_설명.sql` (1절) |
| JVM/Spring | **Flyway** (spring-boot 통합 — 기동 시 또는 `./gradlew flywayMigrate`) | `flyway_schema_history` (Flyway 자체 관리) | `V001__설명.sql` (Flyway 규약 — 1절 순번 원칙은 동일) |
| 기타 | 해당 생태계 표준 도구 채택, 없으면 커스텀 러너를 이식 | 도구 자체 테이블 | 도구 규약 |

- JVM 채택 시 `00_schema_migrations.sql`은 복사하지 않는다 — Flyway의 이력 테이블이 그 역할이다. 멱등성(2절)·DDL/DML 분리(3절)·2단계 배포(4절)는 러너와 무관하게 동일 적용.
- 운영 적용은 배포 절차의 별도 단계로 실행하고(docker.md 5절 — 앱 기동과 분리), 실행 전 백업을 확인한다.
- **canonical DDL 동기화**: 프로젝트에 기초 테이블 DDL 파일(테이블 원본 정의)을 유지한다면, 마이그레이션으로 바뀐 내용을 그 파일에도 반영한다 (근거: 운영 테이블에는 있는 컬럼이 원본 DDL에 없으면, 신규 환경 구축 시 스키마가 달라진다).
- 배포 전 미적용 마이그레이션 확인은 "배포 준비" 명령이 수행한다.
