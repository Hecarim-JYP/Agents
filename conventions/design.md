# 디자인·UI/UX 컨벤션 (JYP)

신규 React 프로젝트의 화면 디자인, 테마, UX 규칙. `react.md`를 전제로 한다.
기존 프로젝트는 그 프로젝트의 기존 스타일링 방식을 유지한다 (혼용 금지).

## 1. 스타일링 스택 (신규 프로젝트 표준)

- **Tailwind CSS + shadcn/ui** 를 표준으로 한다 (2026-07-11 확정).
- 컴포넌트가 필요하면 **직접 만들기 전에 shadcn/ui에 있는지 먼저 확인**한다 (`npx shadcn@latest add table` 등). 테이블·폼·모달·셀렉트·토스트 등 업무 UI 대부분이 검증된 상태로 제공된다.
- 커스텀 컴포넌트를 만들 때도 shadcn 패턴(cva variants, `cn()` 유틸)을 따른다 — 프로젝트 안에서 스타일 작성 방식이 하나여야 한다.
- 별도 CSS 파일 작성은 Tailwind로 표현 불가능한 경우(복잡한 인쇄 양식, keyframe 등)로 한정하고, 사유를 주석으로 남긴다.

## 2. 디자인 토큰 (STRICT)

- **색상은 시맨틱 토큰으로만** 사용한다: `bg-background`, `text-foreground`, `text-muted-foreground`, `bg-primary`, `text-destructive`, `border-border` 등 (shadcn CSS 변수 체계).
- **hex/임의 색상 직접 지정 금지**: `bg-[#3b82f6]`, `text-blue-500` ✗ → `bg-primary` ○ (근거: 색을 직접 박으면 다크모드가 깨지고, 브랜드 색 변경 시 전체 파일을 뒤져야 한다).
- 프로젝트 브랜드 색은 프로젝트 시작 시 `:root` CSS 변수 한 곳에서 정의하고 CLAUDE.md에 기록한다.
- 간격·라운드·그림자는 Tailwind 기본 스케일을 쓴다 (`p-4`, `gap-2`, `rounded-md`). 임의값(`p-[13px]`)은 금지 — 스케일을 벗어나야 할 이유가 있으면 토큰으로 추가.

## 3. 다크모드

- 시맨틱 토큰(2절)을 지키면 다크모드는 CSS 변수 전환으로 자동 대응된다 — 이것이 토큰 강제의 가장 큰 이유.
- 테마 전환은 `class` 전략(`dark:` variant), 초기값은 시스템 설정 따름 + 사용자 토글 저장.
- 업무 시스템은 다크모드가 필수는 아니다 — 지원 여부를 프로젝트 시작 시 결정해 CLAUDE.md에 기록 (단, 토큰 규칙은 지원 여부와 무관하게 지킨다).

## 4. 두 가지 UI 모드

프로젝트 시작 시 모드를 선택해 CLAUDE.md에 기록한다:

| | **업무 시스템 모드 (기본)** | **서비스 모드** |
|---|---|---|
| 목표 | 정보 밀도·작업 효율 | 첫인상·가독성 |
| 간격 | 촘촘하게 (`text-sm`, `p-2`~`p-4`, compact 테이블) | 여유 있게 (`text-base`, `p-6`+, 카드 레이아웃) |
| 화면 구성 | 검색폼 + 데이터 테이블 + 모달 중심 | 히어로/대시보드/카드 중심 |
| 장식 | 최소화 — 그림자·애니메이션 절제 | 절제된 사용 허용 |
| 조작 | 키보드 효율 우선 (Enter 검색, 포커스 순서) | 마우스/터치 우선 |
| 반응형 | 데스크톱 우선 (최소 지원 폭 명시, 예: 1280px) | 모바일 퍼스트 |

## 5. 화면 구성 규칙 (업무 시스템 모드 기준)

- **데이터 테이블**: 숫자는 우측 정렬, 텍스트는 좌측 정렬. 헤더 고정(sticky), 행 hover 표시, 상태는 색+텍스트 뱃지(색만으로 구분 금지). 페이지네이션·컬럼 정렬은 공용 컴포넌트로.
- **폼**: 라벨은 입력 위 또는 좌측으로 프로젝트 내 통일. 필수 항목 표시(`*`), 검증 에러는 **해당 필드 바로 아래**에 구체적으로 ("잘못된 입력" ✗ → "숫자만 입력 가능합니다" ○). 저장/취소 버튼 위치와 순서를 전 화면 통일.
- **상태 표현 3종 필수**: 모든 데이터 영역은 로딩(스켈레톤 또는 공용 Loading)·빈 상태("데이터가 없습니다" + 다음 행동 안내)·에러 상태를 명시적으로 처리한다. 빈 화면을 그냥 두지 않는다.
- **피드백**: 성공은 토스트(흐름 안 끊음), 실패·확인 필요는 다이얼로그. 파괴적 동작(삭제 등)은 반드시 확인 다이얼로그 (react.md 5절 공용 다이얼로그 규칙과 연계).

## 6. 접근성 기본선

- 텍스트 대비 4.5:1 이상 (시맨틱 토큰을 쓰면 대체로 충족).
- 모든 인터랙션 요소는 키보드로 도달·조작 가능해야 하고, 포커스 표시(`focus-visible` 링)를 제거하지 않는다.
- 아이콘 단독 버튼에는 `aria-label`, 의미 있는 이미지에는 `alt` 필수.
- 색상만으로 정보를 전달하지 않는다 (상태 뱃지 = 색 + 텍스트).

## 7. 아이콘·폰트

- 아이콘: **lucide-react** 표준 (shadcn 기본 짝). 프로젝트 내 아이콘 라이브러리 혼용 금지.
- 폰트: 한글 UI는 **Pretendard**(또는 시스템 폰트 스택) 기본. 프로젝트 시작 시 확정해 CLAUDE.md에 기록.

## 8. 셋업 레시피 (Vite React — 스캐폴드가 수행)

"Tailwind + shadcn 셋업"은 아래 산출물이 전부 갖춰져야 완료다 — 조정표 한 줄로 얼버무리지 않는다 (근거: 레시피 없이는 매 프로젝트 수동 셋업에서 항목이 누락된다). 도구 버전별 세부 명령은 shadcn 공식 문서(`ui.shadcn.com/docs/installation/vite`)가 우선 — 아래는 완료 판정 체크리스트다.

1. **Tailwind 설치·배선**: `tailwindcss` 설치 + Vite 플러그인 등록, `index.css`에 Tailwind 지시문.
2. **`@` alias**: `tsconfig.json`(`baseUrl`/`paths`)과 `vite.config.ts`(`resolve.alias`) **양쪽** 설정 — 한쪽만 하면 에디터 또는 빌드가 깨진다.
3. **shadcn init**: `npx shadcn@latest init` → `components.json` 생성. ⚠ JavaScript 프로젝트면 `components.json`의 `"tsx": false` 확인 — 기본값 true라 JS 프로젝트에 .tsx가 생성된다.
4. **시맨틱 토큰**: init이 만든 `:root`/`.dark` CSS 변수 블록을 `index.css`에 유지하고, 브랜드 색(2절)을 여기서 조정.
5. **`cn()` 유틸**: `src/lib/utils`(clsx + tailwind-merge) 생성 확인 — 커스텀 컴포넌트도 이것만 사용 (1절).
6. **다크모드 배선** (지원 프로젝트만, 3절): `class` 전략 ThemeProvider + 토글 컴포넌트 + localStorage 저장 + 시스템 설정 초기값 — 토큰만 있고 토글 배선이 없으면 "지원"이 아니다.
7. **아이콘·폰트**: lucide-react 설치, Pretendard(또는 확정 폰트) 로드 (7절).
8. 검증: shadcn 컴포넌트 하나를 `npx shadcn@latest add button`으로 추가해 빌드가 통과하는지 확인 후, 데모 사용처는 남기지 않는다.
