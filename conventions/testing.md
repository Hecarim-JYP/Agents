# 자동화 테스트 컨벤션

무엇을·어떻게 테스트하고, 테스트 없는 변경을 어떻게 다루는지 정의한다. (2026-07-11 확정 — 근거: 정적 추적/수동 스모크만으로는 회귀 안전망이 없어 "코드완료·런타임 검증 필요" 상태가 누적되는 문제)

## 0. 테스트 DB 사용 여부 — 프로젝트 시작 시 결정 (스캐폴드 체크리스트)

서비스·리포지토리를 어떻게 테스트할지는 **테스트 전용 DB를 두는가**에 따라 갈린다. 시작 시 확정해 CLAUDE.md에 기록한다.

| | (a) 테스트 DB 사용 (권장) | (b) 테스트 DB 없음 |
|---|---|---|
| 서비스 테스트 | **실제 테스트 DB에 연결해 검증** — 쿼리·트랜잭션·제약까지 함께 검증된다 | 쿼리 계층을 **목킹**(Vitest `vi.mock` / Spring `@MockBean`)하고 서비스 로직만 검증 |
| 쿼리 테스트 | 가능 — 동시성(낙관적 락·조건부 UPDATE, sql.md 8절)과 UNIQUE 제약은 **실제 DB에서만 검증된다** | 불가 — 이 영역은 테스트로 못 잡는다는 한계를 CLAUDE.md에 명시 |
| 구성 | dev override의 `db-test` 서비스(스캐폴드 템플릿에 포함 — 데이터는 tmpfs라 기동 시 초기화) + `.env`의 `TEST_DB_*`. 각 테스트는 트랜잭션 롤백 또는 데이터 초기화로 **서로 독립**하게 (4절) | 추가 구성 없음 |

- (a)를 선택하면 **CI에도 테스트 DB를 띄운다** — GitHub Actions job의 `services:` 블록(스캐폴드 test.yml 템플릿에 포함). 이걸 빠뜨리면 로컬은 통과하고 CI만 깨진다. 운영·개발 DB를 테스트에 절대 사용하지 않는다.
- 어느 쪽이든 **인터페이스 추상화(리포지토리 인터페이스 + 구현체 주입)를 위해 계층을 늘리지 않는다** — 테스트 가능성은 실제 DB 또는 모듈 목킹으로 확보하고, 코드 구조는 단순하게 유지한다 (patterns.md 6절 성급한 추상화 방지).

## 1. 무엇을 테스트하나 (우선순위)

1. **서비스 계층 비즈니스 로직 단위 테스트** — 계산, 검증, 분기, 상태 전이. 0절의 결정에 따라 테스트 DB 연결 또는 쿼리 목킹으로 수행한다(순수 계산은 도메인 함수로 분리해 DB 없이 검증 — patterns.md 2절).
2. **API 계약 통합 테스트** — 엔드포인트별 응답 상태코드·응답 키 구조·에러 본문 계약(patterns.md 3절). 계약이 코드로 고정되면 응답 형식 회귀를 자동으로 잡는다.
3. **버그 재발 방지 테스트** — 버그를 수정할 때는 가능하면 **재현 테스트를 먼저** 쓰고 수정한다. 백로그 완료 항목의 "재현 안 됨" 근거가 된다.

## 2. 무엇을 하지 않나

- **커버리지 % 목표 금지** — 수치를 채우기 위한 무의미한 테스트가 양산된다. 위 우선순위 대상만 채운다.
- 단순 매핑/getter/상수 테스트, UI 스냅샷 남발 금지.
- 실행 순서에 의존하는 테스트 금지 — 각 테스트는 독립적으로 통과해야 한다 (general.md 7절).

## 3. 프레임워크 표준

| 스택 | 도구 |
|---|---|
| TypeScript/JavaScript (클라·서버 공통) | **Vitest** (Vite 생태계, 설정 최소) |
| API 통합 테스트 (TS) | Vitest + **supertest** (서버 기동 없이 앱 객체 직접 호출) |
| **Java/Spring** | **JUnit 5** — API 계약은 `@WebMvcTest`/`MockMvc`, 서비스 도메인 로직은 컨텍스트 없는 순수 단위 테스트. 실행은 `./gradlew test` (spring.md 7절). 메서드명은 4절의 Java 예외 참조 |
| React 컴포넌트 | Testing Library — 핵심 인터랙션(권한 게이트, 폼 검증)만 |
| Python | **pytest** |

## 4. 구조와 네이밍

- 위치는 프로젝트 초기에 결정해 CLAUDE.md에 기록: `tests/` 미러링 또는 소스 인접 `*.test.*` 중 하나로 통일.
- 테스트 이름은 검증하는 행동을 서술한다: `만료된_세션은_거부된다`, `생성_성공시_201과_data_키를_반환한다`.
  - ⚠ **Java는 예외 (2026-07-14 실측)**: Checkstyle의 `MethodName` 규칙이 camelCase를 강제하므로 **메서드명은 camelCase로 두고 서술은 `@DisplayName`에 적는다** (`@DisplayName("생성_성공시_201과_data_키를_반환한다") void createReturns201WithData()`). 스네이크 메서드명을 쓰면 `./gradlew check`가 실패한다 — 규칙끼리 부딪히므로 도구 쪽을 따른다.
- 실행 명령은 항상 `npm test`(또는 `pytest`)로 통일 — 스크립트 별칭을 두어 CI와 로컬이 같은 명령을 쓴다.

## 5. 게이트 규칙 (STRICT — 게이트+기록 방식)

- **핵심 로직(서비스 계층·계산·검증·상태 전이·권한 판정) 변경 시 테스트 동반 작성이 원칙.**
- 사정상 못 썼으면 숨기지 말고 기록한다:
  1. 완료 보고에 **"테스트 미작성"** 명시
  2. changelog(dev_log)에 미작성 사실과 사유 명시
  3. 백로그에 `TEST-` ID로 항목 추가 (심각도·근거 포함)
- 검증 단계에서 `npm test`를 실행하고 결과를 보고한다. **실패 테스트를 숨기거나 skip 처리로 얼버무리는 것 금지** — 실패는 실패라고 보고.
- 문서·설정만 바꾼 변경은 게이트 대상이 아니다.
