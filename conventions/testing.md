# 자동화 테스트 컨벤션 (JYP)

무엇을·어떻게 테스트하고, 테스트 없는 변경을 어떻게 다루는지 정의한다. (2026-07-11 확정 — 근거: 정적 추적/수동 스모크만으로는 회귀 안전망이 없어 "코드완료·런타임 검증 필요" 상태가 누적되는 문제)

## 1. 무엇을 테스트하나 (우선순위)

1. **서비스 계층 비즈니스 로직 단위 테스트** — 계산, 검증, 분기, 상태 전이. DB 없이 돌아가야 한다(쿼리는 mock 하거나 순수 함수로 분리).
2. **API 계약 통합 테스트** — 엔드포인트별 응답 상태코드·응답 키 구조·에러 본문 계약(patterns.md 3절). 계약이 코드로 고정되면 응답 형식 회귀를 자동으로 잡는다.
3. **버그 재발 방지 테스트** — 버그를 수정할 때는 가능하면 **재현 테스트를 먼저** 쓰고 수정한다. 백로그 완료 항목의 "재현 안 됨" 근거가 된다.

## 2. 무엇을 하지 않나

- **커버리지 % 목표 금지** — 수치를 채우기 위한 무의미한 테스트가 양산된다. 위 우선순위 대상만 채운다.
- 단순 매핑/getter/상수 테스트, UI 스냅샷 남발 금지.
- 실행 순서에 의존하는 테스트 금지 — 각 테스트는 독립적으로 통과해야 한다 (general.md 7절).

## 3. 프레임워크 표준

| 스택 | 도구 |
|---|---|
| TypeScript/JavaScript (클라·서버 공통) | **Vitest** (Vite 생태계, 설정 최소) |
| API 통합 테스트 | Vitest + **supertest** (서버 기동 없이 앱 객체 직접 호출) |
| React 컴포넌트 | Testing Library — 핵심 인터랙션(권한 게이트, 폼 검증)만 |
| Python | **pytest** |

## 4. 구조와 네이밍

- 위치는 프로젝트 초기에 결정해 CLAUDE.md에 기록: `tests/` 미러링 또는 소스 인접 `*.test.ts` 중 하나로 통일.
- 테스트 이름은 검증하는 행동을 서술한다: `만료된_세션은_거부된다`, `생성_성공시_201과_data_키를_반환한다`.
- 실행 명령은 항상 `npm test`(또는 `pytest`)로 통일 — 스크립트 별칭을 두어 CI와 로컬이 같은 명령을 쓴다.

## 5. 게이트 규칙 (STRICT — 게이트+기록 방식)

- **핵심 로직(서비스 계층·계산·검증·상태 전이·권한 판정) 변경 시 테스트 동반 작성이 원칙.**
- 사정상 못 썼으면 숨기지 말고 기록한다:
  1. 완료 보고에 **"테스트 미작성"** 명시
  2. changelog(dev_log)에 미작성 사실과 사유 명시
  3. 백로그에 `TEST-` ID로 항목 추가 (심각도·근거 포함)
- 검증 단계에서 `npm test`를 실행하고 결과를 보고한다. **실패 테스트를 숨기거나 skip 처리로 얼버무리는 것 금지** — 실패는 실패라고 보고.
- 문서·설정만 바꾼 변경은 게이트 대상이 아니다.
