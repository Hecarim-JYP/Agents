# 배포·운영 컨벤션

배포, 환경 관리, 로깅, 장애 대응, 보안 점검 규칙. 운영까지 가는 모든 프로젝트에 적용한다.

## 1. 환경 분리

- 환경별 설정 파일 분리: `.env.development` / `.env.production` (또는 배포 환경의 환경변수 주입). 코드는 `NODE_ENV`로 분기하지 말고 **설정값으로** 분기한다.
- `.env.example`은 항상 최신으로 유지 — 새 환경변수를 추가하면 같은 커밋에서 example에도 키를 추가한다 (값은 더미). 배포 준비 명령이 이 대조를 수행한다.
- 운영 환경 설정값을 변경하면 **변경 내용·사유·일시를 changelog에 기록**한다 — 설정 변경은 코드 변경만큼 장애 원인이 되지만 git에 안 남는다.
- 시크릿에 하드코딩 폴백 금지, 미설정 시 기동 실패 (patterns.md 4절).
- **시크릿 보관·전달**: 원본은 패스워드 매니저(또는 암호화 저장소) **한 곳**에 보관하고, 보관 위치를 인수인계 문서(handover)에 기록한다. 새 기기·운영 서버 전달은 그 경로로만 — 메신저·메일 평문 전송 금지. `.env` 파일도 암호화 백업 대상에 포함한다 (git에 없으므로 기기가 죽으면 유일본이 사라진다).
- 유출이 의심되면 즉시 해당 시크릿을 로테이션(재발급)하고 incident로 기록한다.

## 2. 배포 원칙

- **main 브랜치 = 항상 배포 가능한 상태.** 작업은 feature/fix 브랜치에서 하고 완료 후 main에 병합한다. 운영 배포는 main에서만.
- 배포 시점에 태그를 남긴다: `v{major}.{minor}.{patch}` (semver) — **major**: 호환성이 깨지는 변경·대규모 개편 / **minor**: 기능 추가 / **patch**: 버그 수정·소규모 개선. 장애 시 "직전 배포가 뭐였는지"를 태그로 즉시 찾는다.
- **배포 단위 = Docker 컨테이너 이미지** (개발 서버·운영 서버 공통, 상세: `docker.md`). 이미지 태그는 git 배포 태그와 연동한다.
- 배포 전 반드시 "배포 준비" 명령(dev-claude)의 체크리스트를 통과한다.
- 배포 직후 최소 확인: 컨테이너 헬스체크·기동 로그 정상, 핵심 화면/엔드포인트 1개 수동 확인. 결과를 changelog에 기록.
- 롤백 계획 없이 배포하지 않는다 — 기본 수단은 **직전 태그 이미지로 재기동**(docker.md 5절) + 미적용 마이그레이션 확인.

## 3. 로깅 (STRICT)

- 로그 레벨을 구분한다: `error`(즉시 대응 필요) / `warn`(비정상이나 동작함) / `info`(운영 이벤트 — 기동, 배포, 주요 처리). 디버그 출력을 운영에 남기지 않는다.
- **에러 로그에는 맥락 필수**: 무슨 작업이었는지, 어떤 입력이었는지, 어느 사용자/요청이었는지. `console.error(err)` 한 줄 금지 — 스택과 함께 맥락을 남긴다.
- 요청 단위 추적: 요청마다 식별자(요청 ID)를 부여해 한 요청의 로그를 이어 볼 수 있게 한다.
- **민감정보(비밀번호, 토큰, 개인정보) 로그 금지.** 요청 본문을 통째로 로깅하지 않는다.
- 컨테이너 환경에서는 stdout/stderr로 출력하고(docker.md 6절) 로깅 드라이버 로테이션으로 영속화한다 — 로그가 유실되면 장애 분석이 불가능하다.

## 4. 장애 대응

- 장애 발생 시 `templates/incident.md` 형식으로 `docs/incidents/YYYY-MM-DD_요약.md`에 기록한다 — 타임라인, 원인, 응급조치/근본조치, 재발 방지.
- 응급 수정(hotfix)도 작업 사이클을 지킨다: 수정 → 배포 → **사후에라도** 작업 정리(changelog·백로그) 수행. "급해서 기록 생략"이 반복되면 시스템 이력이 끊긴다.
- 근본 원인이 즉시 해결 불가능하면 백로그에 High로 등록하고 incident 문서에서 링크한다.

## 5. 오픈 전 보안 체크리스트

첫 운영 배포 전 전 항목 점검, 이후 배포는 변경된 영역만. "배포 준비" 명령이 이 표를 사용한다.

| # | 항목 | 기준 |
|---|---|---|
| 1 | CORS | 화이트리스트 방식 (`origin` 무제한 개방 금지) |
| 2 | 보안 헤더 | helmet(또는 동등 수단) 적용, HTTPS 종단 확인 |
| 3 | 인증 토큰 | 만료 시간이 합리적인가(장기 토큰 금지), 로그아웃/세션 무효화 동작 |
| 4 | 권한 게이트 | 관리/설정 엔드포인트에 역할·권한 미들웨어 부착 확인 (로그인 여부만으로 불충분) |
| 5 | 신뢰값 주입 | 테넌트/행위자 값 서버 강제 주입 동작 확인 (express.md 5절) |
| 6 | 시크릿 | `.env` 커밋 이력 없음, `.gitignore` 적용, 운영 시크릿은 개발과 분리 |
| 7 | 파일 업로드 | 경로 트래버설 방어, 확장자/크기 제한, 업로드 폴더 비추적 |
| 8 | 에러 노출 | 운영 에러 응답에 스택 트레이스·내부 경로 미노출 |
| 9 | 의존성 | `npm audit` high 이상 취약점 0건 (불가피하면 사유 기록) |
| 10 | 컨테이너 | 이미지에 `.env`/시크릿 미포함(`.dockerignore` 확인), **우리가 빌드하는 이미지·프록시 non-root 실행**(벤더 이미지는 벤더 권한 설계 — docker.md 2절, Caddy 채택 시 예외 기록 확인), 운영 이미지 `latest` 태그 미사용 |

## 6. 백업

- 운영 DB는 정기 백업을 설정하고, **복구 절차를 프로젝트 CLAUDE.md에 문서화**한다 (백업이 있어도 복구 방법을 모르면 없는 것과 같다).
- 업로드 파일 등 DB 밖 자산도 백업 대상에 포함한다.
- **복구 리허설 — 분기 1회 (STRICT, 2026-07-13 확정)**: 최신 백업을 개발서버(또는 로컬)에 실제로 복원하고 배포 모드로 기동해 핵심 화면 1개가 동작하는지 확인한다. 결과와 소요 시간을 changelog에 기록, 실패하면 백로그 High 등록 (근거: 이 패턴 운영 사고의 전형이 "백업은 있는데 복구가 안 됨" — 복원해 본 적 없는 백업은 백업이 아니다).

## 7. 가동 모니터링·알림

**운영 개시 전에 모니터링 주체를 결정한다 — 스캐폴드 체크리스트에서 사용자에게 반드시 확인** (사내 인프라팀이 있으면 위임이 기본). 어느 쪽이든 "사용자 제보로 장애를 처음 아는 상태"를 금지한다.

| 선택 | 조치 |
|---|---|
| (a) 사내 인프라팀 위임 (기본) | 헬스 엔드포인트 URL(`/health` 또는 `/actuator/health`)과 장애 시 연락처를 인프라팀에 전달하고, 위임 사실·전달 내용을 CLAUDE.md에 기록 |
| (b) 자체 구축 | Uptime Kuma 같은 경량 도구를 **감시 대상과 다른 호스트**(예: 개발서버에서 운영서버를 감시)에 컨테이너로 상주 + 실패 시 알림 채널(메일/메신저) 연결 — 같은 서버에 두면 서버가 죽을 때 감시도 같이 죽는다 |

## 8. 서버 호스트 관리

컨테이너 아래층(호스트 OS·디스크·Docker 데몬)도 관리 대상이다 — 방치하면 디스크 풀·미패치가 컨테이너 위 규칙을 무력화한다.

- **호스트 OS 관리 주체(인프라팀/직접)를 CLAUDE.md에 기록**한다. 직접 관리라면 보안 패치 주기를 정한다.
- **디스크 점검은 배포 시마다** (deploy-check와 함께): 사용량 확인 + 오래된 이미지 정리. 정리는 `docker system prune -a` 전체 삭제 금지 — 롤백용 직전 태그 이미지(docker.md 5절 "1~2개 보관")가 지워진다. `docker image prune -a --filter "until=720h"` 같이 보관 기간을 두고 정리한다.
- 로그 로테이션은 docker.md 6절(로깅 드라이버 `max-size`/`max-file`) — 설정돼 있는지 첫 배포 시 확인.
