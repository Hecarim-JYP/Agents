# 구현 패턴 — 언어 무관 (JYP)

어떤 스택이든 적용하는 구조·설계 패턴. 스택이 React/Express면 `react.md`/`express.md`를 추가로 적용한다.

## 0. 설계 우선순위 — 확장성·유연성·유지보수성 (2026-07-14 확정)

세 축을 모두 요구하면 "미리 유연하게 만들라"와 "최소한만 만들라"(dev-rules 단순성 규칙)가 충돌하는 것처럼 보인다. **판정 기준은 되돌리기 비용이다.**

### 0-1. 되돌리기 비용으로 가른다 (STRICT)

| | 되돌리기 어려운 결정 | 되돌리기 쉬운 결정 |
|---|---|---|
| 무엇 | DB 스키마, API 계약(필드명 포함 — api.md 4절), 계층 경계, 인증 소스, 멀티테넌트·다국어 여부, 배치 실행 방식 | 내부 구현, 화면 로직, 유틸 구성, 알고리즘 |
| 왜 | 데이터가 쌓이고 클라이언트가 붙은 뒤에는 바꾸는 데 마이그레이션·2단계 배포·전면 수정이 든다 | 나중에 고쳐도 비용이 작다 |
| 원칙 | **시작 시 결정하고 확장 여지를 확보한다** — 스캐폴드 체크리스트가 이 목록이다 | **최소로 구현한다** — 요청받지 않은 유연성·설정화·추상화 금지 |

- **"요청받지 않은 유연성 금지"(dev-rules)는 오른쪽 칸에만 적용된다.** 왼쪽 칸에서 확장 여지를 빠뜨리는 것은 단순함이 아니라 부채다.
- 새로운 "시작 시 결정" 항목을 발견하면 스캐폴드 체크리스트에 추가한다 — 체크리스트는 되돌리기 어려운 결정의 레지스트리다.

### 0-2. 확장은 추상화가 아니라 경계와 표로 확보한다

**미래를 위한 인터페이스·플러그인 구조·설정 스위치를 미리 만들지 않는다.** 이 저장소가 실제로 쓰는 확장 방식은 다음 둘이다:

- **원칙 층 / 구현 층 분리** — 원칙은 스택·DB 무관으로 쓰고, 구현은 표로 매핑한다 (database.md 8절 타 DB 매핑, api.md 5절 봉투 계약, migration.md 5절 러너, auth.md 6절 스택 매핑, integration.md 8절). 새 스택·DB를 채택하면 **표에 열을 추가**하는 것이 확장이다.
- **데이터로 확장** — 상태·구분값은 코드 분기가 아니라 공통코드 테이블로(database.md 4절). 코드 추가 = INSERT.
- 코드 레벨의 공통화는 **세 번째 등장 시 추출**(6절)이 기준이다 — 두 번째까지는 복제 허용.

### 0-3. 다중 인스턴스 가정 (STRICT)

지금은 앱 인스턴스가 1개(docker.md 7절: 서버 1대 = 프로젝트 1개)라도, **앱 프로세스에 상태를 두지 않는다** — 인스턴스를 2개로 늘리는 순간 조용히 깨지는 것들이다:

| 상태 | 잘못된 방식 | 규칙 |
|---|---|---|
| 세션 | 인메모리 세션 | 무상태 토큰(auth.md 3절) — 이미 충족 |
| 업로드 파일 | 컨테이너 로컬 디스크 | 공유 볼륨(docker.md 4절) — 인스턴스 다중화 시 공유 스토리지로 전환 필요 |
| 캐시 | 프로세스 메모리 (권한 캐시 등) | 단일 인스턴스 전제로만 허용 — **CLAUDE.md에 "다중화 시 전환 대상"으로 기록** |
| 스케줄러 | 앱 내 스케줄러 | 별도 배치 컨테이너 (batch.md 1절) |

- 위 표에서 단일 인스턴스에 의존하는 항목을 채택할 때는 **CLAUDE.md의 "다중화 전환 목록"에 기록한다** — 전제가 깨질 때 무엇을 손봐야 하는지가 문서에 남아야 한다.

## 1. 계층 분리

- 흐름은 단방향: **진입점(라우팅/컨트롤러) → 비즈니스(서비스) → 데이터 접근(리포지토리/쿼리)**. 역방향 의존 금지.
- 진입점은 얇게 — 요청 해석과 응답 반환만. 비즈니스 로직을 두지 않는다.
- 데이터 접근 계층은 저장/조회만 — 비즈니스 판단을 두지 않는다. 외부 시스템 호출(사내·외부 API)도 이 계층과 동급의 연동 계층으로 분리하고 서비스가 조율한다 (integration.md).
- **데이터 계층의 "단순" = 무판단이지, SQL이 단순해야 한다는 뜻이 아니다 (STRICT).** 집합 연산(JOIN·집계·필터·정렬·페이징)은 DB가 잘하는 일이므로 쿼리에 둔다. 서비스가 전체 행을 가져와 반복문으로 필터링·조인·집계하는 것 금지 — N+1 쿼리와 함께, 목록 조회가 수십 초씩 걸리는 성능 저하의 최대 원인. 경계 기준: "데이터를 **어떻게 가져오나**"는 쿼리, "가져온 데이터로 **무엇을 판단하나**"(적합 판정, 상태 전이 허용, 권한)는 서비스.

## 2. 책임 배치 (STRICT)

- **입력 검증과 타입 정규화는 비즈니스 계층 책임.** 데이터 계층은 받은 값을 pass-through 한다 (근거: 계층마다 제각각 형변환하면 같은 파라미터가 함수마다 다르게 동작하는 불일치 발생).
- 필수 파라미터 검증은 비즈니스 함수 최상단에서 일괄 수행.
- **서비스 비대화 방지 — 서비스는 "절차의 조율자"로 유지한다.** 서비스가 로직의 주소인 건 맞지만 한 파일일 필요는 없다. 순수 계산(판정 로직, 포맷 규칙, 채번 규칙 등 입력→출력이 명확한 것)은 서비스가 호출하는 **별도 도메인 함수/헬퍼로 분리**한다 — DB·HTTP 없이 단독 테스트 가능해지고(testing.md 1절의 1순위 대상), 서비스 파일이 수천 줄로 자라는 것을 막는다.
- 표시/포맷 로직(소수 자리, 단위, 상태 라벨)은 **단일 출처(SSOT)** — 화면마다 복제하지 않고 한 곳에 두고 가져다 쓴다 (근거: 화면별로 포맷을 복제하면 같은 데이터가 화면마다 다르게 보이는 문제 발생).

## 3. 응답/에러 계약

- API 응답 형식은 **공용 응답 헬퍼로 통일**하고, 응답 키 계약(목록/상세/생성/수정이 각각 어떤 키를 쓰는지)을 문서화한다. 혼용 금지 (근거: 키 혼용은 클라이언트의 응답 접근을 깨뜨린다).
- 에러는 **타입 있는 에러**(ValidationError, NotFoundError, ForbiddenError 등)로 던지고, 중앙 에러 핸들러가 상태코드/응답 본문으로 변환한다. 개별 지점에서 임의 형식으로 응답하지 않는다.
- 성공 판정을 특정 상태코드 하나와 비교하지 않는다 — 2xx 범위 또는 예외 기반으로 판정 (근거: 생성 응답은 보통 201이라 `=== 200` 비교는 신규 등록을 실패 처리한다).

## 4. 신뢰 경계 (STRICT)

- 클라이언트가 보낸 **식별자·권한 관련 값(사용자 ID, 테넌트/회사 ID, 작성자, 역할)은 신뢰하지 않는다.** 서버가 검증한 값(세션/토큰)으로 강제 주입·비교한다 (근거: 클라 전송값을 신뢰하면 타 테넌트 데이터 접근·행위자 스푸핑 같은 보안 결함으로 직결).
- UI에서 버튼을 숨기거나 비활성화하는 것은 UX일 뿐 보안이 아니다 — **서버가 같은 권한 코드로 최종 차단**해야 한다.
- 비밀값·내부 주소는 `환경변수 || '하드코딩 폴백'` 패턴 금지 — 미설정이면 기동 실패로 즉시 드러나게 한다.

## 5. 상태와 데이터 변경

- 진행 상태 전이는 **명시적 이벤트로만** 일어난다. 다른 필드 저장의 부수효과로 상태가 바뀌는 설계 금지 (근거: 날짜 하나 입력했는데 진행 상태가 바뀌어버리는 식의 부수효과 버그).
- **동시 실행 방어는 서비스의 상태 검사가 아니라 데이터 계층의 조건부 갱신이 최종 방어선이다 (STRICT)** — "조회 → 상태 확인 → 갱신"은 확인과 갱신 사이에 틈이 있어 동시 요청이 둘 다 통과한다(이중 승인·이중 발송). 갱신 조건에 현재 상태·버전을 넣고, 변경된 행이 0이면 충돌(409)로 처리한다 (sql.md 8절). 상태 전이의 부수 작업(메일 발송 등)은 갱신이 실제로 1행을 바꾼 뒤에만 수행한다.
- 여러 테이블/자원을 함께 쓰는 변경은 트랜잭션으로 묶고, 실패 시 롤백과 함께 부수 자원(업로드 파일 등)도 정리한다.
- 판별자 성격의 필드(종류/구분값)는 등록 후 변경 불가로 설계한다 — 바꾸면 파생 데이터 정합이 깨진다.

## 6. 공용화 기준

- 같은 UI 패턴/로직이 **세 번째로 등장하면** 공용 컴포넌트·유틸로 추출한다. 두 번째까지는 복제 허용(성급한 추상화 방지).
- 공용화한 뒤에는 개별 구현 재도입을 금지하고, 공용 지점을 프로젝트 CLAUDE.md에 기록한다.
