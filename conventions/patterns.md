# 구현 패턴 — 언어 무관 (JYP)

어떤 스택이든 적용하는 구조·설계 패턴. 스택이 React/Express면 `react.md`/`express.md`를 추가로 적용한다.

## 1. 계층 분리

- 흐름은 단방향: **진입점(라우팅/컨트롤러) → 비즈니스(서비스) → 데이터 접근(리포지토리/쿼리)**. 역방향 의존 금지.
- 진입점은 얇게 — 요청 해석과 응답 반환만. 비즈니스 로직을 두지 않는다.
- 데이터 접근 계층은 저장/조회만 — 비즈니스 판단을 두지 않는다.
- **데이터 계층의 "단순" = 무판단이지, SQL이 단순해야 한다는 뜻이 아니다 (STRICT).** 집합 연산(JOIN·집계·필터·정렬·페이징)은 DB가 잘하는 일이므로 쿼리에 둔다. 서비스가 전체 행을 가져와 반복문으로 필터링·조인·집계하는 것 금지 — N+1 쿼리와 함께 성능 저하의 최대 원인 (근거: LTMS FED-23 — "전체조회 28초"류 성능 미스터리의 전형적 씨앗). 경계 기준: "데이터를 **어떻게 가져오나**"는 쿼리, "가져온 데이터로 **무엇을 판단하나**"(적합 판정, 상태 전이 허용, 권한)는 서비스.

## 2. 책임 배치 (STRICT)

- **입력 검증과 타입 정규화는 비즈니스 계층 책임.** 데이터 계층은 받은 값을 pass-through 한다 (근거: 계층마다 제각각 형변환하면 같은 파라미터가 함수마다 다르게 동작하는 불일치 발생 — LTMS에서 컨벤션으로 확정).
- 필수 파라미터 검증은 비즈니스 함수 최상단에서 일괄 수행.
- **서비스 비대화 방지 — 서비스는 "절차의 조율자"로 유지한다.** 서비스가 로직의 주소인 건 맞지만 한 파일일 필요는 없다. 순수 계산(판정 로직, 포맷 규칙, 채번 규칙 등 입력→출력이 명확한 것)은 서비스가 호출하는 **별도 도메인 함수/헬퍼로 분리**한다 — DB·HTTP 없이 단독 테스트 가능해지고(testing.md 1절의 1순위 대상), 서비스 파일이 수천 줄로 자라는 것을 막는다 (근거: LTMS — 성적서 서비스류 파일 비대화).
- 표시/포맷 로직(소수 자리, 단위, 상태 라벨)은 **단일 출처(SSOT)** — 화면마다 복제하지 않고 한 곳에 두고 가져다 쓴다 (근거: LTMS #45 — 화면별 포맷 복제로 같은 데이터가 화면마다 다르게 보이는 문제).

## 3. 응답/에러 계약

- API 응답 형식은 **공용 응답 헬퍼로 통일**하고, 응답 키 계약(목록/상세/생성/수정이 각각 어떤 키를 쓰는지)을 문서화한다. 혼용 금지 (근거: LTMS #6 — 헬퍼별 키 혼용으로 클라이언트 접근이 깨지는 버그).
- 에러는 **타입 있는 에러**(ValidationError, NotFoundError, ForbiddenError 등)로 던지고, 중앙 에러 핸들러가 상태코드/응답 본문으로 변환한다. 개별 지점에서 임의 형식으로 응답하지 않는다.
- 성공 판정을 특정 상태코드 하나와 비교하지 않는다 — 2xx 범위 또는 예외 기반으로 판정 (근거: LTMS — 생성 응답이 201이라 `=== 200` 비교가 신규 등록을 실패 처리하던 버그 반복).

## 4. 신뢰 경계 (STRICT)

- 클라이언트가 보낸 **식별자·권한 관련 값(사용자 ID, 테넌트/회사 ID, 작성자, 역할)은 신뢰하지 않는다.** 서버가 검증한 값(세션/토큰)으로 강제 주입·비교한다 (근거: LTMS SEC-01/SEC-19 — 클라 전송값 신뢰로 타사 데이터 접근·대리 결재가 가능했던 보안 결함).
- UI에서 버튼을 숨기거나 비활성화하는 것은 UX일 뿐 보안이 아니다 — **서버가 같은 권한 코드로 최종 차단**해야 한다.
- 비밀값·내부 주소는 `환경변수 || '하드코딩 폴백'` 패턴 금지 — 미설정이면 기동 실패로 즉시 드러나게 한다 (근거: LTMS #55).

## 5. 상태와 데이터 변경

- 진행 상태 전이는 **명시적 이벤트로만** 일어난다. 다른 필드 저장의 부수효과로 상태가 바뀌는 설계 금지 (근거: LTMS #49 — 날짜 입력이 상태를 바꿔버리는 부수효과 버그).
- 여러 테이블/자원을 함께 쓰는 변경은 트랜잭션으로 묶고, 실패 시 롤백과 함께 부수 자원(업로드 파일 등)도 정리한다.
- 판별자 성격의 필드(종류/구분값)는 등록 후 변경 불가로 설계한다 — 바꾸면 파생 데이터 정합이 깨진다 (근거: LTMS #61).

## 6. 공용화 기준

- 같은 UI 패턴/로직이 **세 번째로 등장하면** 공용 컴포넌트·유틸로 추출한다. 두 번째까지는 복제 허용(성급한 추상화 방지).
- 공용화한 뒤에는 개별 구현 재도입을 금지하고, 공용 지점을 프로젝트 CLAUDE.md에 기록한다.
