# 다국어(i18n) 컨벤션

다국어 지원 프로젝트에 적용한다. **다국어 여부·지원 언어 목록·기본 언어는 프로젝트 시작 시 결정**해 CLAUDE.md에 기록한다 (스캐폴드 체크리스트 항목) — 하드코딩된 문구를 나중에 키로 소급 추출하는 비용이 매우 크다.

## 1. 공통 원칙

- locale 식별자는 **BCP 47** 소문자 기본형으로 통일: `ko`, `en`, `ja`, `zh` (지역 구분이 실제로 필요할 때만 `zh-tw` 식 확장).
- **사용자에게 보이는 모든 문구는 키로 관리한다 (STRICT)** — 기본 언어(한국어)라고 컴포넌트·서버 코드에 직접 쓰지 않는다 (근거: "기본 언어만 하드코딩" 예외를 허용하면 두 번째 언어 추가 시점에 전체 코드를 뒤져야 한다).
- 키 네이밍: `{module}.{screen}.{name}` (예: `ct.request.title`, `common.button.save`). 공용 문구는 `common.` 네임스페이스.
- 언어 우선순위(fallback 사슬): 사용자 선호 locale → 소속 법인 기본 locale(`company.locale` — 05_company.sql) → 프로젝트 기본 언어. 어느 층에서든 번역이 없으면 다음 층이 아니라 **기본 언어 문구로 fallback**하고, 키 원문을 화면에 노출하지 않는다.
- **사용자 입력 데이터는 번역 대상이 아니다** — 입력된 언어 그대로 저장·표시한다. 번역 대상은 UI 문구와 코드성 데이터(3절)뿐.

## 2. 클라이언트 (React)

- 표준 도구: **react-i18next** (+ `i18next-browser-languagedetector`). 리소스는 `src/shared/locales/{locale}/{namespace}.json` — 네임스페이스는 모듈 단위로 분리해 화면별 지연 로드가 가능하게 한다.
- 언어 전환은 사용자 토글 + 선택값 저장(localStorage 허용 — 토큰이 아닌 UI 선호값이므로 auth.md 3절 금지 대상이 아니다). 초기값은 로그인 사용자의 선호 locale.
- 날짜·숫자·통화 표시는 번역 파일에 포맷을 심지 말고 **`Intl.DateTimeFormat`/`Intl.NumberFormat`(현재 locale)** 사용 — api.md 4절(포맷팅은 클라이언트)의 다국어 구현.
- 복수형·변수 삽입은 i18next 문법(`{{count}}`, plural)로 처리 — 문자열 연결로 문장을 조립하지 않는다 (근거: 언어마다 어순이 달라 연결 조립은 반드시 깨진다).

## 3. 서버·DB

- **에러 분기의 SSOT는 `error` 코드다** (api.md 5절) — 클라이언트는 코드를 키로 번역해 표시하고, 봉투의 `message`는 fallback 문구로 취급한다. 서버가 요청 언어(`Accept-Language` 또는 사용자 선호값)로 `message`를 만들 수 있으면 더 좋지만, 코드 없이 message 문자열에만 의존하는 분기·표시는 금지.
- 서버가 직접 발신하는 산출물(메일, 알림, PDF/출력물)은 클라이언트 번역을 탈 수 없다 — 수신자의 선호 locale로 서버 측 번역을 적용한다 (리소스는 클라이언트와 같은 키 체계).
- **코드성 데이터의 표시명은 번역 테이블로**: 원본 테이블에 언어별 컬럼(`name_ko`, `name_en`)을 늘리는 방식 금지 (근거: 언어 추가마다 DDL 변경 + 전 쿼리 분기 수정). 표준 패턴:

```sql
CREATE TABLE IF NOT EXISTS common_code_i18n (
    common_code_i18n_id  BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    common_code_id       BIGINT UNSIGNED NOT NULL,
    locale               VARCHAR(10)  NOT NULL,          -- BCP 47
    code_name            VARCHAR(100) NOT NULL,          -- 해당 언어 표시명
    created_at           DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    created_by           BIGINT,
    updated_at           DATETIME NULL ON UPDATE CURRENT_TIMESTAMP,
    updated_by           BIGINT,
    UNIQUE KEY uq_common_code_i18n (common_code_id, locale),
    KEY idx_common_code_i18n_locale (locale)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

- 원본 테이블의 표시명 컬럼(`code_name` 등)은 기본 언어 값으로 유지한다 — 번역 누락 시 fallback이자 관리 화면의 기준값.
- 조회는 `LEFT JOIN {테이블}_i18n ON ... AND locale = :locale` + `COALESCE(i18n.code_name, 원본.code_name)` 패턴 — 번역 누락이 빈 값이 아니라 기본 언어로 나온다.

## 4. 사용자 선호 locale

- `user` 테이블에 `locale VARCHAR(10) NULL` 컬럼 추가 (NULL이면 법인 기본값 사용 — 1절 fallback 사슬). 로그인 응답 또는 내 정보 API로 클라이언트에 전달한다.
- 시간대는 locale과 별개 결정이다 — database.md 1절(시간대 통일)을 따르고, 다국가 시간대가 실제 요구되면 저장은 UTC로 전환하고 CLAUDE.md에 기록한다.
