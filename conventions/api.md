# API 설계 규칙 (JYP)

REST API 엔드포인트 설계 표준. 응답 봉투·에러 계약은 이 문서 5절(스택 무관), 인증은 auth.md.

## 1. URL 설계

- **모든 업무 API는 `/api` 프리픽스 아래에 둔다 (STRICT)** — 프록시(배포)와 Vite proxy(로컬)가 `/api/*`만 서버로 라우팅하는 동일 출처 구성의 전제다 (docker.md 7절). 프리픽스 밖에 업무 라우트를 만들면 프록시가 정적 파일로 취급해 404가 난다. 아래 예시 경로는 `/api` 이후 기준 (`GET /requests` = 실제 `GET /api/requests`).
- 소문자, 리소스 중심 명사. 컬렉션은 복수형: `GET /requests`, 단건은 `GET /requests/:id`.
- 계층은 소속 관계일 때만: `/requests/:id/items`. 3단계 이상 중첩 금지 — 필요하면 쿼리 파라미터로.
- 동사는 CRUD로 표현 불가능한 액션에만 예외 허용: `POST /requests/:id/approve`.
- 사내 시스템은 버전 프리픽스 없이 시작. 외부 공개 API가 되면 그때 `/v1` 도입.

## 2. HTTP 메서드·상태코드

| 동작 | 메서드 | 성공 코드 |
|---|---|---|
| 목록/상세 조회 | GET (부수효과 금지) | 200 |
| 생성 | POST | **201** |
| 수정 | PUT/PATCH | 200 |
| 삭제 (소프트삭제 포함) | DELETE | 200 또는 204 |

- 에러: 400(검증 실패) / 401(미인증) / 403(권한 없음) / 404(없음) / 409(중복·충돌) / 500(서버 오류). 검증 실패를 200 + 에러 메시지로 반환하지 않는다.
- **409는 동시성 충돌의 표준 응답이다** (sql.md 8절): 낙관적 락 실패("다른 사용자가 먼저 수정했습니다 — 새로고침 후 재시도"), 이미 처리된 상태 전이("이미 승인된 건입니다"). 조회 응답에 `version`을 포함하는 리소스는 수정 요청에 `version`을 받는다.
- 클라이언트 성공 판정은 2xx 범위/예외 기반 (react.md 3절 — `=== 200` 단독 비교 금지의 서버측 짝).

## 3. 목록 조회 표준 파라미터

- 페이징: `?page=1&size=20` — 응답에 `total`(전체 건수) 포함. 페이징 없는 전체 조회는 상한을 정하거나 명시적으로 설계 결정.
- 정렬: `?sort=created_at,desc` (복수 정렬은 반복 지정).
- 검색·필터: 명시적 쿼리 파라미터 (`?status=DONE&keyword=...`). 필터 조합 로직은 서비스 계층에서.

## 4. 필드·형식

- **API 필드명 = DB 컬럼명과 동일한 snake_case로 통일** — 중간 변환 계층을 두지 않는다 (근거: camelCase 변환층이 있으면 계층 경계마다 이름이 달라져 추적 비용과 매핑 버그가 생긴다).
- ⚠ **그 대가로 컬럼명이 API 계약의 일부가 된다 (STRICT)**: 응답에 실리는 컬럼의 이름 변경·타입 변경·삭제는 **API 파괴적 변경으로 취급**한다 — 마이그레이션만 돌리면 되는 일이 아니라 클라이언트가 깨진다. migration.md 4절의 **2단계 배포**(새 컬럼 추가 → 클라이언트 전환 → 구 컬럼 삭제)를 적용하고, 변경을 changelog에 기록한다. 내부 전용 컬럼(응답에 싣지 않는 것)은 이 제약을 받지 않는다.
- 날짜는 `YYYY-MM-DD`, 일시는 ISO 8601 형식으로 통일. 포맷팅(표시용 변환)은 클라이언트에서.
- 같은 리소스는 어느 엔드포인트에서든 같은 필드명·구조로 반환한다 (목록/상세의 축약 차이는 허용하되 이름은 동일하게).

## 5. 응답 봉투·에러 계약 (스택 무관, STRICT)

**봉투는 API의 계약이지 프레임워크의 기능이 아니다** — Express든 Spring이든 클라이언트가 받는 JSON 형태는 동일해야 한다 (근거: 계약이 서버 스택에 묶이면 스택 교체·혼용 시 클라이언트 전체가 깨진다). 프로젝트 시작 시 확정해 CLAUDE.md에 기록하고, 프로젝트 내 혼용 금지.

기본 계약 (신규 프로젝트 표준):

```jsonc
// 성공 — 모든 응답이 단일 봉투: data 키 하나
{ "data": { ... } }                       // 상세/생성/수정
{ "data": [ ... ], "total": 42 }          // 목록 (페이징 시 total 필수 — 3절)

// 에러 — 상태코드(2절)와 함께 고정 키
{ "message": "사용자에게 보여줄 메시지", "error": "VALIDATION_ERROR", "field": "email" }
```

- `message`는 사용자 표시용(한국어/현재 언어), `error`는 클라이언트 분기용 코드(대문자 스네이크), `field`는 검증 실패 필드(해당 시).
- 개별 엔드포인트에서 임의 형식으로 응답하지 않는다 — 스택별 공통 구현을 반드시 거친다 (patterns.md 3절).

**스택별 구현 층** (계약은 위와 동일, 도구만 다르다):

| 스택 | 성공 봉투 | 에러 변환 |
|---|---|---|
| Express | 공용 응답 헬퍼 (express.md 2절) | 중앙 errorHandler 미들웨어 |
| Spring | 공통 래퍼 + `ResponseEntity` (spring.md 2절) | `@RestControllerAdvice` (spring.md 4절) |
