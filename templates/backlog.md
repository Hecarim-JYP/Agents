# 개선·리팩토링 백로그

> 최초작성: YYYY-MM-DD [박진영]
> 용도: 이력관리 + 리팩토링 착수 시 우선순위/상태 추적.
> 이 문서는 한 번 쓰고 버리는 문서가 아니라 **상시 최신화하는 살아있는 단일 출처(SSOT)** 다.

## 운영 규칙

- **발견 즉시 추가**: 작업(버그수정/리팩토링/리뷰/분석) 중 새 개선·결함을 발견하면 지금 당장 고치지 않더라도 그 자리에서 항목을 추가한다. 누락 방지가 목적.
- **ID 체계**: 카테고리 접두사 + 카테고리 내 일련번호 (예: `SEC` 보안 / `BE` 백엔드 / `FE` 프론트 / `DB` DB·마이그레이션 / `TEST` 테스트 미작성·부재 / `OPS` 배포·운영 / `DEAD` 죽은코드 — 프로젝트에 맞게 정의). 기존 ID 재사용·재번호 금지.
- **필수 필드**: 심각도(High/Med/Low) · 작업량(S/M/L) · 상태 · 근거(`파일:라인`) · 문제 · 제안. **근거 없는 추측 항목 금지.**
- **⚠ 발견 ≠ 완료 (STRICT)**: 어떤 항목이 발견·언급·관련되었다는 사실만으로 절대 ☑(완료) 처리 금지. ☑는 ① 항목의 "제안"이 실제 코드에 반영됐고 ② `git diff`로 확인되는 변경 파일 근거가 있을 때만. 일부만 처리했으면 `◐ 진행중`(무엇이 남았는지 메모). 근거 없이 완료 처리하면 SSOT가 깨져 실제 미처리 작업이 누락된다 — 가장 위험한 실수.
- **완료 처리**: `☑ (YYYY-MM-DD)` + 변경 파일·한 줄 요약. P0 요약표에 있으면 표도 함께 갱신. 연관 동반 수정도 기록.
- **오탐/정상 확인**은 부록 A에, **문서-코드 불일치**는 부록 B에 기록해 재작업을 방지한다.

## 상태 범례

`☐ 예정` · `◐ 진행중` · `☑ 완료` · `⊘ 보류(WontFix)`
심각도: **High**(정합/보안/기능) · **Med**(구조/성능 부채) · **Low**(정리/일관성). 작업량: S(시간) · M(반나절~1일) · L(수일).
