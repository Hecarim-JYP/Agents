# 작업 정리

> ⚠ 작업 단위 = 현재 브랜치의 **커밋되지 않은 변경 전체** (`git status` + `git diff HEAD`, untracked 신규파일 포함). "이번 세션에서 건드린 파일"만 정리하는 것 금지 — 이전 세션의 미커밋 변경도 전부 포함. 이미 커밋된 변경은 제외.

아래 5단계를 순서대로 수행한다:

1. **changelog 작성/갱신** — `docs/dev_log/YYYY-MM-DD_CHANGELOG.md` (날짜 = 오늘, 형식: `~/.claude/jyp/templates/changelog.md`). 같은 날짜 파일이 있으면 새로 만들지 말고 누적/갱신. 테스트 미작성 핵심 로직 변경이 있으면 그 사실과 사유를 명시.
2. **memory 최신화** — 이번 작업에서 확정된 사용자 선호·프로젝트 사실을 메모리에 반영. 기존 메모리와 중복이면 해당 파일을 갱신. 코드/git 히스토리로 알 수 있는 내용은 저장하지 않는다.
3. **CLAUDE.md 갱신** — 새로 생긴 컨벤션·Pitfall·아키텍처 변경 반영. 기존 항목과 겹치면 새 번호를 추가하지 말고 기존 항목을 갱신.
4. **백로그 동기화** — `docs/REFACTORING_BACKLOG.md`의 예정(☐) 항목 전체와 미커밋 변경을 대조. **발견 ≠ 완료 (STRICT)**: 실제 해결됐고 `git diff` 근거가 있는 것만 `☑ (YYYY-MM-DD)`, 일부 처리는 `◐`, 새로 발견한 이슈는 항목 추가 (형식: `~/.claude/jyp/templates/backlog.md`). 백로그 파일이 없으면 건너뛴다.
5. **요약 보고** — 무엇을 어디에 기록했는지, 백로그 대조 결과(완료 N건/미해결 M건) 명시. 커밋/푸시는 사용자가 명시적으로 요청할 때만.
